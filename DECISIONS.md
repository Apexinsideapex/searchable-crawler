# Decisions Log

Deviations from the phase plans, and why.

## Phase 1 — Foundation

- **Email confirmation left ON** in Supabase Auth (plan recommended disabling it for a frictionless demo signup). User chose to keep the default. Consequence: `signup` doesn't auto-create a session — the UI shows "check your email" and the user must click the confirmation link before logging in. RLS/integration tests bypass this by creating pre-confirmed users via the Admin API (`email_confirm: true`).
- **`middleware.ts` renamed to `proxy.ts`** with the exported function renamed `middleware` → `proxy`. Next.js 16.2.10 deprecated the `middleware` file convention in favor of `proxy`; using the old name emitted a build warning (and would eventually break). Behavior is unchanged — same route matching, same session-refresh logic in `lib/supabase/middleware.ts`.
- **Vercel deploy done via CLI**, not GitHub-triggered CI. Project linked with `vercel link`, env vars (`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`) set for Production + Development via `vercel env add`. Preview-environment vars were not set (Vercel required an explicit branch and it's not needed yet) — add them before relying on preview deployments.
- **`SUPABASE_SERVICE_ROLE_KEY` and DB password live only in `.env.local`** (gitignored) and were not pushed to Vercel, per the plan — no Vercel server route uses the service role yet. They'll be needed as Supabase Edge Function secrets in Phase 2, not as Vercel env vars.

## Phase 6.1 — Onboarding (partial, done ahead of the rest of Phase 6)

- **Scope**: only 6.1 (add-site flow, install/settings page, send-test-crawl) was pulled forward, plus the PRD FR-3.2/AC-2 platform-breakdown chart gap discovered while comparing Phase 3/4 against the PRD (it was never actually covered by any phase — 04-dashboard-mvp.md's own chart spec is category-stacked, not platform-grouped, despite its AC-2 checklist line implying a platform chart existed). 6.2 (visual polish), 6.3 (README), and 6.4 (demo rehearsal) are still outstanding.
- **"Send test crawl" has no live green flip.** The 06 plan calls for the verification button to flip green via a realtime subscription the moment the event lands — that infrastructure is Phase 5's live feed, not built yet. Shipped instead: a manual "Send test crawl" → manual "Check now" pair (`app/dashboard/actions.ts`: `sendTestCrawl` posts a synthetic GPTBot event through the real deployed `ingest` Edge Function tagged `source: verification, is_verification: true`; `checkVerificationEvent` reads it back via a 5-minute time-window query). Functionally proves AC-6 end to end, just via a button press instead of a push. Revisit once Phase 5 adds the realtime channel — swap the poll for a subscription.
- **`platform_breakdown` RPC** (`supabase/migrations/0003_platform_breakdown.sql`) added as a fourth `security invoker` dashboard RPC, same RLS-delegation pattern as the three in `0002`. Ranks platforms by event count within the filtered window, capped at top 10.
- **Settings page CSV export is a real button, fixed to a 30d/all-categories/all-platforms window.** Initially shipped as just a link back to the dashboard's existing `CsvExportButton`; user asked for an actual working export on Settings instead. Reused the same `CsvExportButton` component (no new export logic) with a hardcoded `{ range: "30d", category: "all", platforms: null }` filter object, since Settings has no `FilterBar` to derive filters from. A note in the card points back to the dashboard for a range/category-specific export. `hasEvents` is computed via a `fetchStatsSummary` call for that fixed window.
- **App origin for the pixel snippet is hardcoded** (`https://searchable-crawler.vercel.app`) in `app/dashboard/[siteId]/settings/page.tsx`, matching the existing hardcoded convention in `README.md` / `proxy.ts` rather than introducing a new env var for a single-Vercel-project app.
